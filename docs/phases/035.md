# Title [PF-304]: Implement Vendor Payment Methods List Endpoint (`GET /vendor/payment-methods`)

## Type

Feature

## Description

Vendors need the ability to retrieve the payment methods currently stored on file for their store. These payment methods are stored in the `payment_methods` table and represent cards vaulted with Square and mirrored internally.

This endpoint will allow the vendor dashboard to display all stored payment methods (currently cards, future support for ACH).

The endpoint must retrieve all payment methods associated with the authenticated vendor store and return normalized DTOs consistent with the existing `vendorPaymentMethodResponse` structure.

## Scope

Included:

* Add **GET `/vendor/payment-methods`** endpoint
* Resolve vendor `storeID` using `vendorcontext.ResolveVendorStoreID`
* Fetch payment methods using `billing.Repository.ListPaymentMethodsByStore`
* Map DB models → response DTO
* Return ordered list (created_at DESC already enforced in repo)
* Response shape aligned with existing `vendorPaymentMethodResponse`

Explicitly NOT included:

* Pagination
* Filtering
* ACH support
* Square API calls

## Acceptance Criteria

* [ ] `GET /vendor/payment-methods` route exists under vendor route group
* [ ] Endpoint resolves vendor store ID via `vendorcontext`
* [ ] Endpoint returns all payment methods for that store
* [ ] Response includes `id`, `card_brand`, `card_last4`, `card_exp_month`, `card_exp_year`, `is_default`, `created_at`
* [ ] Endpoint returns empty array when no payment methods exist
* [ ] Endpoint returns `200 OK` with `data.payment_methods`

## Dependencies

Blocked by:

* Existing `PaymentMethod` GORM model

Blocks:

* Vendor billing UI for payment method management

## Technical Notes

Relevant components:

Controller:

```
api/controllers/billing/vendor_payment_methods_list.go
```

Service layer:

```
internal/paymentmethods/service_list.go
```

Repository usage:

```
billing.Repository.ListPaymentMethodsByStore(ctx, storeID)
```

DTO mapping recommended via mapper file:

```
internal/paymentmethods/mapper.go
```

Response structure:

```
{
  "data": {
    "payment_methods": []
  }
}
```

ASSUMPTION:
Existing repository method `ListPaymentMethodsByStore` should be reused without modification.

## Out of Scope

* Square card synchronization
* Payment method pagination
* ACH payment method types

# Title [PF-305]: Implement Vendor Payment Method Deletion (`DELETE /vendor/payment-methods/:id`)

## Type

Feature

## Description

Vendors must be able to remove stored payment methods from their store.

This endpoint will delete a payment method stored in the `payment_methods` table belonging to the vendor's store.

Deletion removes the local database record. Handling of Square-side deletion is intentionally excluded for this phase.

## Scope

Included:

* Add **DELETE `/vendor/payment-methods/:id`**
* Resolve vendor `storeID`
* Validate payment method belongs to the store
* Delete payment method record

Explicitly NOT included:

* Deleting payment method in Square
* Reassigning default payment method
* Cascading billing updates

## Acceptance Criteria

* [ ] Endpoint exists under vendor routes
* [ ] Endpoint resolves vendor store ID
* [ ] Endpoint validates the payment method belongs to the store
* [ ] Endpoint deletes the record from `payment_methods`
* [ ] Endpoint returns `204 No Content` on success
* [ ] Returns `404` if payment method does not exist or does not belong to store

## Dependencies

Blocked by:

* PaymentMethod model
* Billing repository

Blocks:

* Vendor billing UI delete action

## Technical Notes

Controller:

```
api/controllers/billing/vendor_payment_method_delete.go
```

Repository additions required:

```
DeletePaymentMethod(ctx context.Context, storeID uuid.UUID, paymentMethodID uuid.UUID)
```

Recommended DB query:

```
DELETE FROM payment_methods
WHERE id = ? AND store_id = ?
```

ASSUMPTION:
Deleting a default payment method will **not automatically promote another method** in this ticket.

## Out of Scope

* Square card deletion
* Default reassignment logic
* Billing plan impact

# Title [PF-306]: Implement Toggle Default Payment Method (`PATCH /vendor/payment-methods/:id`)

## Type

Feature

## Description

Vendors must be able to designate one stored payment method as the default.

Setting a payment method as default should automatically clear the existing default for the store to ensure only one default exists.

This logic mirrors the behavior already used when storing a card (`ClearDefaultPaymentMethod`).

## Scope

Included:

* Add **PATCH `/vendor/payment-methods/:id`**
* Payload to toggle `is_default`
* If `true`, clear previous default using `ClearDefaultPaymentMethod`
* Update target payment method

Explicitly NOT included:

* Updating other payment method fields
* Square API updates

## Acceptance Criteria

* [ ] Endpoint exists under vendor routes
* [ ] Payload allows `{ is_default: true }`
* [ ] Endpoint validates payment method belongs to store
* [ ] Setting a method as default clears previous default
* [ ] Only one payment method can be default per store
* [ ] Endpoint returns updated payment method DTO

## Dependencies

Blocked by:

* Billing repository
* Existing `ClearDefaultPaymentMethod`

Blocks:

* Vendor billing UI default toggle

## Technical Notes

Controller:

```
api/controllers/billing/vendor_payment_method_update.go
```

Service layer:

```
internal/paymentmethods/service_update.go
```

Repository addition required:

```
UpdatePaymentMethodDefault(ctx context.Context, storeID uuid.UUID, id uuid.UUID, isDefault bool)
```

Logic:

```
tx:
  ClearDefaultPaymentMethod(storeID)
  Update target method -> is_default = true
```

ASSUMPTION:
Endpoint only supports toggling default state (not general updates).

## Out of Scope

* Updating card metadata
* Updating Square data
* Multi-default support

# Title [PF-307]: Extend Store `/store/me` DTO to Include Square Customer ID for Vendors

## Type

Feature

## Description

The vendor dashboard requires access to the Square customer ID associated with the store to support future billing flows and client-side Square integrations.

The `/store/me` endpoint must return the `square_customer_id` for vendor stores.

If the store has no Square customer ID, the response must return an empty string.

## Scope

Included:

* Extend Store DTO returned by `/store/me`
* Add field `square_customer_id`
* Populate from `Store.SquareCustomerID`

Explicitly NOT included:

* Creating Square customers
* Updating Square customer data

## Acceptance Criteria

* [ ] `/store/me` response includes `square_customer_id`
* [ ] Field returns store's `SquareCustomerID`
* [ ] Returns empty string if nil
* [ ] Field is only returned for vendor store types
* [ ] Existing fields in `/store/me` remain unchanged

## Dependencies

Blocked by:

* Store model
* Store controller

Blocks:

* Vendor billing initialization flows

## Technical Notes

Modify DTO:

```
api/controllers/store/dto_store_me.go
```

Mapping logic:

```
squareCustomerID := ""
if store.SquareCustomerID != nil {
    squareCustomerID = *store.SquareCustomerID
}
```

Return field:

```
square_customer_id
```

ASSUMPTION:
`/store/me` already resolves the authenticated store and returns store metadata.

## Out of Scope

* Creating Square customers
* Billing onboarding logic
* Square API calls
