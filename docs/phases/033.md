# Title [PF-292]: Implement GET /v1/stores/{store_id} Controller (Authenticated Viewer Access)

## Type

Feature

## Description

Create a new authenticated route `GET /v1/stores/{store_id}` that returns a `StoreDTO` for viewing a store profile (buyer or vendor perspective).

This endpoint enables users to view other stores’ public profile data while shopping or browsing. Unlike `/v1/stores/me`, this route must not assume tenant ownership and must not enforce ownership-based restrictions.

The endpoint must reuse existing service and repository logic where possible (same underlying data retrieval as `StoreProfile`), but adjust authorization logic to allow any authenticated user to view any store.

---

## Scope

* Add new route under `/v1/stores`
* Implement controller handler `StorePublicProfile`
* Require valid JWT authentication (same middleware as existing protected routes)
* Use existing `storeService` to fetch store by ID
* Return `StoreDTO`
* Log request and errors consistently with existing controllers
* Return appropriate HTTP errors (404 if not found, 403 only if explicitly required)

### Explicitly NOT included

* Creating new service layer logic (must reuse existing service/repo)
* Modifying existing `/me` endpoint behavior
* Adding filtering, search, or pagination
* Anonymous access (authentication is still required)

---

## Acceptance Criteria

* [ ] `GET /v1/stores/{store_id}` route is registered under `/v1/stores`
* [ ] Route requires valid JWT authentication
* [ ] Given valid store_id and authenticated user, endpoint returns 200 with full `StoreDTO`
* [ ] If store does not exist, endpoint returns 404
* [ ] Endpoint does NOT require user to belong to the store
* [ ] Response structure matches `StoreDTO` exactly
* [ ] Controller follows existing logging + error handling conventions

---

## Dependencies

* Blocked by:

  * Existing `storeService` and repository must already support fetching store by ID
* Blocks:

  * Store browsing UI
  * Marketplace / shopping views

---

## Technical Notes

* Route registration example:

```go
r.Route("/v1/stores", func(r chi.Router) {
    r.Get("/{storeId}", controllers.StorePublicProfile(storeService, logg))
})
```

* Controller should:

  * Extract `storeId` from URL param
  * Parse UUID
  * Extract authenticated user from context (for future audit logging)
  * Call existing service method (likely same as StoreProfile but parameterized by ID)
  * Return `StoreDTO`

* Must reuse:

  * Existing StoreDTO mapping logic
  * Existing repo query logic
  * Existing error wrapper pattern

* Do NOT duplicate mapping logic in controller.

### ASSUMPTION

* A service method exists or can be reused to fetch store by ID without ownership enforcement.
* KYCStatus, SubscriptionActive, and Licenses are safe to expose to authenticated users.

---

## Out of Scope

* Store visibility rules (e.g., hiding inactive stores)
* Subscription-based gating
* Caching layer
* Audit logging persistence
* Role-based restrictions beyond authentication

---

# Title [PF-295]: Add vendor-scoped public product browse route by store slug (paginated)

## Type

Feature

## Description

Implement a new authenticated “browse vendor products” route that mirrors the shape/behavior of the existing public `GET /api/v1/products` browse endpoint, but scopes results to a single vendor store identified by its slug (from the storefront URL).

This allows the client to load a vendor storefront’s full catalog (paginated) using the vendor slug → vendor store lookup, then fetch all products for that vendor store.

---

## Scope

* Add a new authenticated GET endpoint under `/api` routes to browse products for a specific vendor store
* Endpoint accepts a `vendorSlug` path param (storeId from URL)
* Resolve `vendorSlug` → `vendorStoreId` using existing store service/repo
* Query products filtered by `vendorStoreId`
* Return the same response shape as the existing browse list routes (same shape as `GET /api/v1/products` and vendor inventory list)
* Support pagination (must be consistent with current browse/list pagination approach)

### Explicitly NOT included

* Anonymous access (must remain authenticated under `/api`)
* Changes to existing `GET /api/v1/products` behavior
* New search engine integration (e.g., Elasticsearch) if not already used here
* New product DTO shape (must reuse existing list DTO/mapper)
* Sorting/filtering beyond what the existing browse route already supports (unless already part of the standard query contract)

---

## Acceptance Criteria

* [ ] A new authenticated GET route exists that accepts a vendor storeId and returns paginated products for that vendor store
* [ ] Route resolves storeId to a store and returns 404 if no store matches the storeId
* [ ] If the resolved store is not a vendor store, route returns 400 or 409 (consistent with existing error semantics)
* [ ] Response product list shape matches existing product browse/list shape used by `GET /api/v1/products`
* [ ] Pagination parameters work as expected (same defaults/limits as existing browse/list endpoints)
* [ ] Results are filtered to the resolved vendor store only (no cross-store leakage)

---

## Dependencies

* Blocked by:

  * Store storeId resolution support in stores service/repo (find store by storeId)
  * Product repository/service supports listing by store/vendor store id with pagination
* Blocks:

  * Vendor storefront catalog page client hook

---

## Technical Notes

* Route shape recommendation (store resource owns vendor catalog):

  * `GET /api/v1/stores/{storeId}/products` if your routing conventions already treat `{storeId}` unambiguously.

* Keep it inside the authenticated `/api` group to inherit:

  * `middleware.Auth(...)`
  * `middleware.RateLimit()`
  * (StoreContext is present in the group; do NOT use StoreContext store id for filtering here—filter must use vendor store resolved from storeId)

* Implementation outline:

  1. Controller parses `storeId`
  2. `storeService` fetch store by storeId
  3. Validate store type is vendor
  4. Call `productService` (or repo) list method with `vendorStoreId` + pagination args
  5. Return same DTO shape as browse list

### ASSUMPTION

* Store IDs are unique and there is an existing or planned repo method (e.g., `storesRepo.FindByStoreId`) used elsewhere.
* “Vendor inventory list” and “public browse products” already share a common DTO shape and mapper that can be reused for this route.
* Existing pagination uses either `(page, limit)` or `(cursor, limit)`; this route must match whichever is currently standard.

---

## Out of Scope

* Faceted filtering (strain, category, potency, etc.) unless already supported by the existing browse endpoint contract
* Full-text search improvements
* Caching
* Any front-end route/hook implementation


# Title [PF-297]: Refactor GET /v1/media to Support Canonical Cursor Pagination Response

## Type

Feature

## Description

The existing `GET /v1/media` endpoint currently returns a `ListResult` containing `items` and a single `cursor` string.

We need to extend this endpoint to:

1. Accept canonical pagination inputs from the client (`cursor`, `limit`, optional `page` if required by shared DTO contract).
2. Return a standardized pagination object matching the `ProductPagination` structure:

   * `page`
   * `total`
   * `current`
   * `first`
   * `last`
   * `prev`
   * `next`
3. Replace the current `ListResult` response shape with a new DTO aligned with the Product-style pagination contract.

This change ensures pagination consistency across domains (products, wishlist, media) and allows frontend components to consume a unified pagination schema.

This is a modification of the existing route. No new route should be created.

---

## Scope

### Included

* Modify `controllers.MediaList` to:

  * Accept `cursor` from query params.
  * Continue accepting `limit`, `kind`, `status`, `mime_type`, and `search`.
* Extend `media.ListParams` usage to ensure `Cursor` is passed properly (already embedded via `pkgpagination.Params`).
* Modify `media.Service.ListMedia` to:

  * Return total count.
  * Compute `first`, `last`, `prev`, and `next` cursors.
  * Construct canonical pagination metadata.
* Replace `ListResult` with a new response DTO:

  ```go
  type MediaPagination struct {
      Page    int    `json:"page"`
      Total   int    `json:"total"`
      Current string `json:"current,omitempty"`
      First   string `json:"first,omitempty"`
      Last    string `json:"last,omitempty"`
      Prev    string `json:"prev,omitempty"`
      Next    string `json:"next,omitempty"`
  }

  type MediaListResult struct {
      Items      []ListItem      `json:"items"`
      Pagination MediaPagination `json:"pagination"`
  }
  ```
* Implement repository-level:

  * `Count(ctx, storeID, filters...)`
  * Boundary cursor helpers (similar to wishlist example pattern).
* Preserve cursor-based pagination semantics (created_at DESC, id DESC).
* Maintain signed URL behavior.

### Explicitly NOT Included

* Changing media DB schema.
* Changing sort order.
* Changing filter semantics.
* Migrating media to offset-based pagination.
* Modifying `/presign` or `/delete`.

---

## Acceptance Criteria

* [ ] `GET /v1/media` accepts `cursor` query param and correctly forwards to service layer.
* [ ] Response no longer returns `{ items, cursor }`.
* [ ] Response returns:

  ```json
  {
    "items": [...],
    "pagination": {
      "page": 1,
      "total": 123,
      "current": "...",
      "first": "...",
      "last": "...",
      "prev": "...",
      "next": "..."
    }
  }
  ```
* [ ] `total` reflects accurate store-scoped filtered count.
* [ ] `next` cursor exists only when additional rows exist.
* [ ] `prev` equals incoming cursor when cursor is present.
* [ ] `first` and `last` cursors are computed from dataset boundaries.
* [ ] Pagination logic mirrors wishlist example semantics.
* [ ] No regression in filtering (`kind`, `status`, `mime_type`, `search`).
* [ ] SignedURL behavior remains unchanged.

---

## Dependencies

* Blocked by: None
* Blocks: Frontend migration to canonical pagination structure

---

## Technical Notes

### Relevant Files

* `controllers/media.go`
* `internal/media/service.go`
* `internal/media/repository.go`
* `internal/media/types.go`
* `pkg/pagination`

### Required Additions

1. Repository:

   * `CountMedia(ctx, storeID, filters...)`
   * `FetchMediaBoundaryCursor(ctx, storeID, filters..., first bool)`
   * These must respect same filters as `List`.

2. Service:

   * Normalize limit using `pkgpagination.NormalizeLimit`.
   * Use buffer via `LimitWithBuffer`.
   * Decode incoming cursor via `pkgpagination.ParseCursor`.
   * Encode outgoing cursor via `pkgpagination.EncodeCursor`.

3. DTO Refactor:

   * Remove old `ListResult`.
   * Introduce `MediaPagination`.
   * Introduce `MediaListResult`.

4. Page Field:

   * Since pagination is cursor-based, `Page` will default to `1`.

   **ASSUMPTION**: Page number is cosmetic for UI parity and does not represent offset-based indexing.

5. Cursor Semantics:

   * Maintain ordering: `created_at DESC, id DESC`.
   * Cursor encoding must match `pkgpagination.Cursor`.

---

## Out of Scope

* Offset pagination.
* Infinite scroll UI changes.
* Global pagination abstraction refactor.
* Adding HATEOAS link objects.
* Any DB migration.
* Performance optimization beyond count query addition.

---

If desired, this can be further split into two tickets:

1. Repository + Service refactor.
2. Controller + DTO contract refactor.

For now, this is scoped as one independently actionable feature ticket.
