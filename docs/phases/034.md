# Title [PF-298]: Create Ads Core Tables + GORM Models + Goose Migrations (ads, creatives, placements, rollups)

## Type

Feature

## Description

Introduce the foundational **Postgres source-of-truth schema** for the PackFinderz Ads Engine per Phase 19: Ads/Creatives/Bids/Budgets/State + daily rollups + idempotent usage charges support.

This ticket creates the required tables via **Goose migrations**, adds **GORM models**, and enforces core invariants (unique constraints, indexes, enums where applicable) needed for API CRUD, serving eligibility queries, rollups, and billing.

## Scope

- **Included**
  - Create Goose migrations for:
    - `ads` (campaign-ish entity: status, placement, target_type/target_id, bid_cents, daily_budget_cents, start/end window, store_id owner)
    - `ad_creatives` (media binding + destination URL + metadata)
    - `ad_daily_rollups` (unique `ad_id + day`, counts/spend snapshot)

  - Add/confirm required pq.StringArray columns for attribution (if not already in DB):
    - `cart_records.ad_tokens` pq.StringArray
    - `vendor_orders.ad_token` string
    - `vendor_order_line_items.ad_token` pq.StringArray

  - Ensure `usage_charges` uniqueness supports idempotent daily ad spend settlement:
    - unique `(store_id, type, for_date)` (or equivalent)

  - Add indexes supporting:
    - eligibility query: `(status, placement, start_at, end_at, store_id)`
    - targeting: `(target_type, target_id)`
    - rollups: `(ad_id, day)` unique + lookup index
    - JSONB GIN indexes are optional and must be justified

- **Explicitly NOT included**
  - Serving endpoints, token issuance, Redis pacing, BigQuery inserts, Square charging logic
  - Backfilling data or running production migrations outside normal deploy workflow

## Acceptance Criteria

- [ ] Goose migrations exist and apply cleanly on a fresh DB and an existing DB (as applicable).
- [ ] Tables `ads`, `ad_creatives`, `ad_daily_rollups` exist with correct constraints and indexes.
- [ ] Attribution JSONB columns exist (or migrations are no-ops if already present), and the schema matches Phase 19.
- [ ] `usage_charges` has a uniqueness constraint that enables idempotent daily ad spend charging.
- [ ] GORM models compile and map to the schema with correct types, tags, and relations.

## Dependencies

- Blocked by: None
- Blocks:
  - PF-XXX Ads CRUD APIs
  - PF-XXX Ads Serve/Track engine
  - PF-XXX Daily rollup + billing cron job

## Technical Notes

- Tables / suggested columns (final names can match existing conventions):
  - `ads`
    - `id (uuid pk)`
    - `store_id (uuid)` (advertiser/vendor store)
    - `status` (draft|active|paused|exhausted|expired|archived) **ASSUMPTION**: status stored as string enum IF NOT create the enum
    - `placement` (e.g., hero_home, etc.)
    - `target_type` (store|product)
    - `target_id (uuid)`
    - `bid_cents (int)` CPM bid cents
    - `daily_budget_cents (int)`
    - `starts_at (timestamptz nullable)`
    - `ends_at (timestamptz nullable)`
    - standard timestamps

  - `ad_creatives`
    - `id (uuid pk)`
    - `ad_id (uuid fk -> ads.id)`
    - `media_id (uuid fk -> media.id)` (or soft ref)
    - `destination_url (text)`
    - `headline/body (text nullable)` **ASSUMPTION**
    - timestamps

  - `ad_daily_rollups`
    - `id (uuid pk)` or composite unique
    - `ad_id (uuid fk)`
    - `day (date)`
    - `impressions (int)`
    - `clicks (int)`
    - `spend_cents (int)`

- **ASSUMPTION**: `media` table exists and `media_id` is used for creatives; multiple media IDs per ad can be represented as multiple `ad_creatives` rows.

# Title [PF-301]: Ads Serve + Tracking Engine — Signed Tokens + /ads/serve, /ads/impression, /ads/click (Redis Counters + Dedupe)

## Type

Feature

## Description

Implement the request-time Ads engine:

- Serve endpoint selects a winning eligible ad for a placement request.
- Server issues signed attribution tokens (view + click).
- Impression endpoint validates token and increments Redis counters with dedupe.
- Click endpoint validates token, increments Redis, and redirects.

This creates the server-truth counters used for pacing and billing while enabling client-carried attribution tokens.

## Scope

- **Included**
  - Token signing + verification utility per Phase 19:
    - payload fields: token_id, ad_id, creative_id, placement, target_type, target_id, buyer_store_id, event_type, occurred_at, expires_at, request_id/nonce, signature
    - strict validation: signature, expiry, buyer store matches JWT, enums valid

  - Redis key schema + helpers:
    - counters: `imps:<adId>:<YYYYMMDD>`, `clicks:<adId>:<YYYYMMDD>`, `spend:<adId>:<YYYYMMDD>`
    - dedupe: `impdedupe:<request_id>:<ad_id>:<placement>`, `clickdedupe:<request_id>:<ad_id>`

  - Serving:
    - `GET /v1/ads/serve?placement=...` returns creative + tokens + request_id
    - candidate selection from Postgres repo + Redis budget gate
    - winner selection: highest bid wins + deterministic tiebreak

  - Tracking:
    - `POST /v1/ads/impression` {view_token, request_id} → validate + SETNX dedupe + INCR imps + INCRBYFLOAT spend
    - `GET /v1/ads/click?token=...&request_id=...` → validate + SETNX + INCR clicks + 302 redirect

- **Explicitly NOT included**
  - Checkout attribution materialization
  - Daily rollup job
  - Square charging
  - BigQuery consumer insertion (those are separate tickets)

## Acceptance Criteria

- [ ] `/v1/ads/serve` returns no ad when Redis unavailable (fail closed) and logs structured reason.
- [ ] Tokens are rejected when invalid/expired/buyer store mismatch; counters are not incremented.
- [ ] Dedupe prevents duplicate imps/clicks for same request_id/ad_id(/placement).
- [ ] Spend increments use CPM rule: `bid_cents / 1000` per impression (stored in Redis spend counter).
- [ ] `/v1/ads/click` redirects to destination URL on success and increments click counter once.

## Dependencies

- Blocked by:
  - PF-XXX Ads CRUD API (needs ads to exist)
  - Redis client availability in service wiring

- Blocks:
  - PF-XXX Checkout token validation + attribution stamping
  - PF-XXX Daily rollup + billing

## Technical Notes

- **ASSUMPTION**: Token format is JWT HS256 or HMAC-signed JSON; pick one and standardize in `pkg/ads/token` (or similar).
- Winner selection deterministic tie-break per spec:
  - `hash(request_id + placement + YYYYMMDD + ad_id)` lowest wins.

- Redis outage behavior must be fail-closed for serve/track.

## Out of Scope

- Multi-slot serve responses
- Fraud detection beyond dedupe guards

# Title [PF-XXX]: Daily Ads Settlement Cron — Roll Redis Counters into Postgres (ad_daily_rollups + usage_charges) + Emit Outbox Events

## Type

Feature

## Description

Implement nightly deterministic settlement:

- Read Redis counters for each active ad for Day N
- Upsert `ad_daily_rollups` and insert idempotent `usage_charges`
- Emit an outbox event `ad_spend_rolled_up` (or `ad_daily_charge_recorded`) for downstream BigQuery and billing bridge

This produces the auditable Postgres receipts and the billing artifacts without charging per-event in realtime.

## Scope

- **Included**
  - New cron binary (e.g., `cmd/ads-rollup-daily`)
  - Settlement algorithm:
    - for each ad/day: read `imps`, `clicks`, `spend` from Redis
    - upsert `ad_daily_rollups(ad_id, day)`
    - insert `usage_charges(store_id, type=ad_spend, for_date=day, amount_cents=round(spend))`
      - idempotent via unique constraint

    - emit outbox event after successful TX

  - Rounding policy helper for spend float → cents
  - Fail-safe behavior:
    - if Redis unavailable, job fails and does not write partial PG rows

- **Explicitly NOT included**
  - Executing actual Square charges (separate ticket)
  - BigQuery consumer code (separate ticket)

## Acceptance Criteria

- [ ] Running the job twice for the same day is idempotent (no double charges).
- [ ] Rollups reflect Redis counters for that day with correct rounding.
- [ ] Outbox event is emitted only after PG transaction commits successfully.
- [ ] Job logs structured per-ad settlement outcomes and errors.

## Dependencies

- Blocked by:
  - PF-XXX Create Ads Core Tables + rollups table + usage_charges uniqueness
  - Redis key schema from Serve/Track ticket

- Blocks:
  - PF-XXX Billing (Square) daily charging
  - PF-XXX Analytics consumer rollup inserts to BigQuery

## Technical Notes

- **ASSUMPTION**: job runs “nightly” São Paulo time; the effective “day” is computed in UTC or a configured timezone—must be explicit in code.
- Use `FOR UPDATE SKIP LOCKED` patterns only if iterating DB rows concurrently.

## Out of Scope
